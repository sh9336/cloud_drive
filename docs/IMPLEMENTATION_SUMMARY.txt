📊 TEMPLATE STRUCTURE
─────────────────────────────────────────────────────────

Current default_dir_tree_template.json has 3 folders:

1. uploads
   └─ User-uploaded files
   └─ can_upload: true
   └─ allow_files: true

2. assets
   └─ Static assets and resources
   └─ can_upload: true
   └─ allow_files: true

3. schedules
   └─ Schedule and sync configuration
   └─ can_upload: true
   └─ allow_files: true

🔄 HOW TO ADD NEW FOLDER
─────────────────────────────────────────────────────────

1. Edit: default_dir_tree_template.json
2. Add node with:
   - path: "unique_name"
   - type: "folder"
   - permissions: { can_upload: true, allow_files: true }
3. Restart: make docker-restart
4. Done! Clients can now use "upload_to": "unique_name"

💡 KEY DESIGN PRINCIPLES
─────────────────────────────────────────────────────────

1. Template is Authoritative
   └─ All upload paths defined in template

2. No Database Dependency
   └─ Template loaded at startup, validation in-memory

3. Tenant Agnostic
   └─ Same template for all tenants

4. Zero DB Queries
   └─ Path validation doesn't touch database

5. Version Controlled
   └─ Template stored in Git

6. Easy to Extend
   └─ Add future permission fields without breaking API

📈 PERFORMANCE CHARACTERISTICS
─────────────────────────────────────────────────────────

Template Loading:
  └─ One-time at startup: ~1ms

Path Validation:
  └─ O(1) in-memory lookup: <0.1ms per request

S3 Path Resolution:
  └─ String formatting: <0.1ms per request

Database Impact:
  └─ Zero queries for path validation
  └─ Only DB hit: save file record after validation

📝 USAGE EXAMPLE
─────────────────────────────────────────────────────────

Request:
  POST /api/v1/files/upload-url
  
  {
    "filename": "Resume.pdf",
    "file_size": 1024000,
    "mime_type": "application/pdf",
    "upload_to": "uploads"
  }

Response:
  {
    "success": true,
    "data": {
      "file_id": "550e8400-...",
      "upload_url": "https://s3.aws/...",
      "s3_key": "tenants/abc-123/uploads/550e8400.pdf",
      "expires_in": 900
    }
  }

Invalid Request:
  {
    "upload_to": "invalid_folder"
  }

Error Response:
  {
    "success": false,
    "error": "invalid upload destination: invalid upload destination: invalid_folder"
  }
